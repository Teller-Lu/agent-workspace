# 权限系统（L0 级）

> 本文件定义工作区中所有文件的权限、审批、审计和多终端修改规则。
> 权限判定分**两条线**，都以**画像**（对话姿态）为第一维：
> - **文件写线**：画像 × **L0/L1/L2**（文件多重要）× **范围**（目标是否在对话 cwd 内）→ §二 / §二·补
> - **工具类线**（v0.0.3 新增）：画像 × **工具类**（本地只读 / 联网只读 / 代码执行 / MCP）——管"跑命令 / 联网 / 调 MCP" → §二·补二
>
> 读则另由 **S/P** 管（§八）。
> AI 每次会话开始时应首先读取本工作区 `CLAUDE.md`，仅在权限制度变化时再读本文件。
> 文件路径以当前工作区根目录为准。

---

## 一、基本原则

1. 工作区允许通过 git 在多台终端间同步文件，但所有修改必须可追溯。
2. 默认使用中文交流和生成中文文档。例外：代码、命令、API 名称、标准英文文件名、英文项目目录名可保留英文。
3. 所有路径说明优先使用"工作区根目录"表达，避免硬编码某一台终端的绝对路径。
4. `审计日志.jsonl` 只追加，不修改历史；面向人查询用 `安全审核.ps1`。
5. **审计写入由 hooks 自动完成**，不依赖 AI 自觉调用 `安全审核.ps1`。

## 二、文件权限分级

### 2.1 文件内容修改权限

| 级别 | 名称 | 修改规则 |
|:--:|------|------|
| **L0** | 权限系统 | 需两次用户确认 + 自动备份 + git tag + 审计 |
| **L1** | 指令锁 | 需用户批准 + 审计 |
| **L2** | 正常 | 开发流程中自由修改，hooks 自动审计 |

### 2.2 文件操作权限（新增 / 删除 / 移动）

| 操作 | 级别 | 规则 |
|------|:--:|------|
| 新增文件 | L1 | 需用户批准 + 审计 |
| 删除文件 | L0 | 需两次用户确认（不终止对话） + 自动备份 + git tag |
| 移动/重命名文件 | L0 | 需两次用户确认 + 自动备份 + git tag |

> 删除和移动为 L0 级别，但仅需对话内双次确认，无需终止或重启会话。

### 2.3 生命周期升降级

- 版本开发阶段：该版本的需求/设计/入口文件为 L2
- 版本标记为"已完成"后：自动升级为 L1
- 版本标记为"迭代中"后：降回 L2

## 二·补、画像层（读 permission_mode，与 L 级正交叠加）

> L0/L1/L2 定"这个文件多重要"；**画像**定"当前对话什么姿态"。两者叠加、再乘上**范围**（目标是否在对话 cwd 子树内），共同决定每次写的 allow/ask/deny。

画像共 5 档（纯英文）：**read / search / work / auto / bypass**，由 hook 读 payload 的 `permission_mode` 推断、读 `cwd` 定范围。完整"L 级 × 画像 × 范围"判定表见 `md/画像映射表.md §四`。要点：

- **read / search**（plan 模式）：一切写 deny（只读姿态；要写先切画像）。
- **work**（manual）：范围内 L2 放行、L1/L0 询问；范围外升一档严。
- **auto**（acceptEdits）：范围内 L1/L2 放行、L0 走 ask（人在即便忙别的对话也可确认放行，无人应答即 deny）；范围外一律 deny。
- **bypass**：全放开——唯 S 读禁（§八）仍是不下线地板。

因此本文 §四各流程里的"L0 两次确认 / L1 一次确认"是 **work 画像**下的交互形态；auto 画像下 L0 同样走 ask（有人可确认、真无人应答则 deny），read/search 画像下这些写直接 deny。

> **read 与 search 在 Claude 侧实现上分不出**：两者都映射到 `plan`，hook 拿到的 `permission_mode` 只有一个值。实际统一按 **search** 处理（联网只读放行）。"离线只读"在 Claude Code 里没有对应的原生模式，做不到。

## 二·补二、工具类层（画像 × 工具类，管"跑命令 / 联网 / 调 MCP"）

> **v0.0.3 新增：画像的射程从"只管文件写"扩到全部四类工具。**
> 此前 hook 只挂在 `Edit|Write|NotebookEdit` 上——**Bash / WebFetch / WebSearch / MCP 完全在画像管辖之外**，一律落回 Claude Code 原生白名单。而白名单与 hooks 同理**不继承父目录**（见 §3.6），子文件夹会话白名单归零 → **弹窗轰炸**：每个 Bash 命令、每个网页域名、每个 MCP 工具都要重新弹窗批准，连 `echo`、空命令 `:`、`[ ! -s file ]` 都在弹。更糟的是大量调用来自 subagent，用户看不到 subagent 上下文，只见一个个孤零零的弹窗。

工具按**可逆性 + 危害面**分四类（**不按名字分**），各有天花板：

| 工具类 | 谁 | read / search | work | auto | bypass |
|---|---|---|---|---|---|
| **A 本地只读** | Read / Glob / Grep | allow | allow | allow | allow |
| **B 联网只读** | WebFetch / WebSearch | **allow** | **allow** | **allow** | allow |
| **C 代码执行** | Bash、MCP 的 exec 档 | 纯净只读 allow / 其余 **deny** | 纯净只读 allow / 其余 **divide** | 同 work | allow |
| **D MCP·只读** | search / get / list / navigate… | **allow** | **allow** | **allow** | allow |
| **D MCP·写** | create / update / **delete**… | deny | **ask**（⚠白名单免疫不了） | **ask** | allow |
| **D MCP·未登记** | 未在分级表登记的 | deny | **ask**（保守兜底） | **ask** | allow |

### `divide`（分流）——第四种决策

hook 有四种决策，**语义完全不同，绝不可混用**：

| 决策 | 效果 |
|---|---|
| `allow` | **覆盖白名单**，直接执行，**永不弹窗** |
| `deny` | **覆盖白名单**，直接拒绝 |
| `ask` | **强制弹窗**——⚠ **连白名单里的也照问**，比 divide **更严**，绝不是"温和" |
| **`divide`（分流）** | hook **不表态**（`exit 0` 无输出）→ 落回 Claude Code 原生流程：**查白名单，命中即执行 / 未命中弹窗 ask**；work 的 ask 一直等人，auto 的 ask 无人应答即 deny |

**`divide` 既不是放行、也不是拒绝**——它把决定权**分流**给「白名单 + 人」。

### C 类为什么天花板是 divide（永不 allow）

`Bash(python -)` 的代码是从 **stdin** 喂进去的，**任何基于命令行文本的白名单/规则都看不见它**。所以代码执行类坚决不开 allow 的口子——**不要去手搓命令危险性解析器**（那只会造出一个比原生更差、还自以为安全的东西），而是复用 Claude Code 原生那套声明式、可审计的白名单。

**唯一例外——"纯净只读命令"直放**（专治"被 `echo` 弹窗打断"的荒谬）：

- 命令头 ∈ `{echo, :, [, test, export, command, pwd, ls}`，或形如 `git status|log|diff|show`
- **且**整条命令**不含任何**拼接符 / 重定向 / 命令替换 / 换行：`|` `;` `&` `<` `>` 反引号 `$(` 回车
- 含任一拼接符 → **一律不放**（`echo x && rm -rf /` 也是 echo 开头）

### 画像相对原生白名单的增量价值

原生白名单一旦点了"总是允许"就**永久放行、不管那操作多危险**（某个 `delete_item` 类 MCP 工具点一次"总是允许"，以后删数据**再也不问**）。画像能 **override** 它，两端收口：

| 画像动作 | 干了原生白名单干不了的事 |
|---|---|
| read/search 下 `deny` | 把整类工具**关死**，白名单再宽也没用 |
| **MCP 写类强制 `ask`** | **让白名单免疫不了不可逆操作**——哪怕手滑点过"总是允许"，删除仍每次问 |
| bypass `allow` | 全放 |

中间那一大片 `divide` 交给白名单 + 人——**不跟原生抢活**。

### 横切兜底（不看画像，谁都拦）

Bash 命令触及 S 级路径 → **deny**（`check_permission.ps1` 自查 + `check_s_level.ps1` 双保险，**不依赖多 hook 决策优先级**）。

**MCP 分档**：见 `md/MCP工具分级表.md`（L1）——hook **直接读它**。模板里它是**空壳**，部署后按你实际接入的 MCP 填写；表里没有的按工具名模式兜底（exec → write → readonly，从严到宽）；**模式也匹配不上的一律 ask**，绝不默认放行。

### 3.6 各层级目录的 `.claude/settings.json`（钩子覆盖件）

**成因**：Claude Code 只从"对话工作目录自身的 `.claude`"加载配置、**不回溯父目录**。官方原文：

> Hooks **and other `settings.json` keys** load from the current working directory's `.claude/` folder with **no parent-directory fallback**

这句话管**两件事**（v0.0.2 时只读走了前半句，v0.0.3 补齐）：

1. **hooks 不继承** → 以子文件夹为 cwd 开的会话不加载根钩子 → 绕过门禁 + 审计 + S 隔离。
2. **`permissions` 白名单也不继承** → 子文件夹会话白名单归零，每个 Bash / WebFetch / MCP 调用都要重新弹窗批准 → **弹窗轰炸**（见 §二·补二）。

**对策（部署要求）**：从根的下一层起、直到各叶子项目的**每个可能被当作 cwd 打开的目录**，各放一份 `.claude/settings.json`，注册根 `hooks/` 的同一套脚本（绝对路径；脚本自行算出工作区根）。逻辑单一来源在根 `hooks/`，这些只是"开关"——直接复制根 `.claude/settings.json` 即可。

> **白名单那一半不靠"铺白名单"解决** —— Bash 命令是无穷组合，白名单永远补不完（连 `echo` 都会弹）。改由**画像接管 Bash / Web / MCP**（§二·补二）。

**级别**：根的 `.claude/settings.json` 仍 **L0**（优先命中）；其余各层"钩子覆盖件" **L1**（`check_permission.ps1` / `after_edit.ps1` / `安全审核.ps1` 三个脚本都有 `*\.claude\settings.json → L1` 规则）；`.claude/settings.local.json`（本地放行名单，非钩子）仍 **L2**。

> **⚠ `.gitignore` 必须写 `**/.claude/settings.local.json`（带 `**/`）**：不带 `**` 只匹配根级那一份，各子目录的白名单**全部漏网**。而 `settings.local.json` 里是用户点"总是允许"累积的规则——含**本机绝对路径、邮箱、常访问域名**（= 个人画像），属敏感信息，绝不能入库。

**维护**：新增子项目 / 新增会被当作 cwd 打开的目录 → 照样补一份。

## 四、自动化操作流程（hooks 接管）

### 4.1 L2 修改流程（最快，hooks 全自动）

```
① AI 调用 Edit/Write 工具修改文件
② PreToolUse hook 自动触发，查询文件级别 → L2，直接放行
③ AI 完成修改
④ PostToolUse hook 自动触发：
   ├─ 追加一条 JSON 到 审计日志.jsonl
   └─ git add <文件> && git commit -m "[审计ID] L2 ..."
```

### 4.2 L1 修改流程（一次确认）

```
① AI 调用 Edit/Write 工具修改 L1 文件
② PreToolUse hook 触发，识别为 L1 → 暂停 + 提示用户
③ 用户输入批准指令
④ hook 放行
⑤ AI 完成修改
⑥ PostToolUse hook 自动写审计 + git commit
```

### 4.3 L0 修改流程（两次确认 + 加固）

```
① AI 调用 Edit/Write 工具修改 L0 文件
② PreToolUse hook 触发，识别为 L0
   ├─ 复制原文件到 .backups/L0/{文件名}.{时间戳}.bak
   └─ git tag pre-L0-{时间戳}-{文件简称}
③ 第一次提示用户确认
④ 用户输入第一次批准
⑤ 第二次独立提示
⑥ 用户输入第二次批准
⑦ hook 放行
⑧ AI 完成修改
⑨ PostToolUse hook 自动写审计 + git commit（commit message 含 [L0] 标记）
```

### 4.4 批准指令

用户输入以下任一指令视为批准：`yes` / `同意` / `是` / `y` / `Y`
以下指令视为拒绝：`no` / `拒绝` / `否` / `n` / `N`

### 4.5 用户手动修改文件

用户绕过 AI 直接编辑文件时，**hooks 不会触发**。用户应：
1. 手动 `git add <文件> && git commit -m "[审计ID] ..."`

---

### 4.6 手动批准流（无 hook 的 Agent，如 Codex）

Claude Code 有 PreToolUse/PostToolUse hook，权限自动门禁。**无 hook 的 Agent** 用 安全审核.ps1 手动批准流（同一套策略）：

```
① 安全审核.ps1 -Request -File "路径" [-Reason "原因"]   L0 自动备份+打 tag，记 REQUEST
② 用户：安全审核.ps1 -Approve -File "路径"              记 APPROVE
③ AI 修改文件
④ 安全审核.ps1 -Log -File "路径" -Summary "摘要"        记 MODIFY
```

两种机制写同一份 审计日志.jsonl（schema 一致），查询统一用 -Report。

## 六、多终端修改规则

1. 不同终端通过 git 同步；冲突时以**语义正确**为准，不盲信任一方时间戳。
2. 审计日志按时间线追加，不因终端不同而分段；commit message 的 `[审计ID]` 是跨终端追踪锚点。
3. 终端硬件/账号/代理环境如需登记，使用者可在 `md/` 自建登记文档（本框架默认单终端，不强制）。

## 七、维护规则

1. 权限制度变化 → 更新 §二、§二·补、§二·补二、§三，同时同步**三个持有权限清单的脚本**（`hooks/check_permission.ps1`、`hooks/after_edit.ps1`、`安全审核.ps1`）的清单数组（`L0_files` / `L1_files` / `L1_dirs`），三份须逐字一致。
2. 新增子项目 / 新增会被当作 cwd 打开的目录 → 照 §3.6 补一份 `.claude/settings.json`（否则该目录会话不触发门禁 + 审计 + S 隔离）。
3. 接入新 MCP server → 把它的工具按三档（只读 / 写 / 代码执行）补进 `md/MCP工具分级表.md`（hook **直接读它**）；**拿不准就别写进 readonly**——走兜底 ask 更安全。
4. 工具类矩阵变化 → 同步 §二·补二、`md/画像映射表.md`、`hooks/check_permission.ps1` 头注释的决定表。
5. 本文件 L0，修改需两次用户确认 + 自动备份 + git tag（由 hooks 自动加固）。

## 八、阅读权限分级（S/P 等级）

> S/P 是**阅读维度**的权限，与 L0/L1/L2（修改维度）正交。
> L0/L1/L2 管"能不能改、改了要什么审批"；S/P 管"能不能读"。

### 8.1 等级定义

| 等级 | 含义 | 行为 |
|---|---|---|
| **S（Secret 秘密级）** | 涉密材料 | 拥有方 AI 可读可改；**其他 AI 不可读**（触发时直接 deny，无需人工介入） |
| **P（Public 公开级）** | 一般文件 | 所有 AI 都可自由阅读 |

### 8.2 归属规则

- S 级文件**属于各自工作区**：Codex 工作区的 S 级文件，Codex（拥有方）可读写，Claude 不可读；反之亦然。
- 一般文件默认为 P 级。
- 只有涉密材料（如公司标准、商业机密）才设为 S 级。

### 8.3 .gitignore 规则（必须）

**所有 S 级文件/目录必须列入 `.gitignore`，确保涉密材料永不提交到 git。**

每个工作区的 `.gitignore` 必须包含一个 `S 级文件` 段落，列出该工作区内所有 S 级路径。示例：

```gitignore
# === S 级文件（秘密级，不可入库）===
# 所有标记为 S 级的文件/目录必须在此列出
confidential/
```

### 8.4 执行机制

**check_s_level.ps1**（PreToolUse hook）：拦截 Read（Claude）和 Bash（Claude + Codex）工具中对 S 级路径的访问。

| AI | Read 工具拦截 | Bash 命令拦截 | OS 级读禁 | 配置位置 |
|---|---|---|---|---|
| **Claude** | PreToolUse 匹配 Read → deny ✅ | PreToolUse 匹配 Bash → check_s_level.ps1 ✅ | sandbox.filesystem.denyRead（仅 macOS/Linux） | `.claude/settings.json` |
| **Codex** | 无 Read 工具 | PreToolUse 匹配 Bash → check_s_level.ps1 ✅ | **不支持**（allow-list 模型，无 deny 条目） | `.codex/hooks.json` |

**hook 脚本工作流程**：
1. 从 `md/S级清单.md` 的 `` ```paths `` 代码块读取对方 AI 的 S 级路径
2. 归一化为绝对路径，生成正斜杠/反斜杠/双反斜杠三种变体
3. 检查命令/路径是否包含任何 S 级路径变体
4. 命中 → `permissionDecision: "deny"`（硬拦，无需人工）
5. 未命中 → exit 0（放行）

**已知盲区**：
- 自动注入上下文（CLAUDE.md / memory / SessionStart）不是 Read/Bash 调用 → 拦不到。**约定**：S 级文件不得放入会被自动加载的位置。
- Codex sandbox 不支持 per-path denyRead → 只能靠 hook + .gitignore + 约定（见 `md/S级清单.md`）。

### 8.5 S 级清单管理

- S 级路径清单在 `md/S级清单.md` 的 ` ```paths ` 代码块中维护（每行一路径，`#` 注释）；`check_s_level.ps1` 从这里读取。
- **S 清单 ≠ .gitignore**：S 路径需**同时**列入 `.gitignore`（保证不入库），但 .gitignore 还含日志/临时等非 S 项，两者分开维护、保持同步。
- 变更 S 级标记时，同步更新 `md/S级清单.md` 与 `.gitignore`。

### 8.6 实施说明

S/P 跨 AI 读取拦截由 `check_s_level.ps1`（PreToolUse on `Read|Bash`）从 `md/S级清单.md` 读对方 S 路径、命中即 deny（见 §8.4）。
