# Claude 画像映射表

> 画像是横跨 Claude 与 Codex 的**统一风险姿态词汇**——用户只记这一套名字，各家怎么翻译成自己的原生权限设置是内部实现。
> **画像名统一用英文**（`read / search / work / auto / bypass`），中文只作一句话说明。
> 权属：Claude 侧维护。设计要点：统一命名 + `permission_mode` 驱动机制 + 5 档 + read/search 硬 deny + bypass 保留 S 地板。

## 一、五档画像（唯一要记的一套名字，统一英文）

命名原则：**名字描述"能干啥"，不模仿任何一家原生名**（Claude/Codex 原生名天生不一样，无法一名对两家；对应关系放到 §二的对照卡）。

| 画像 | 一句话说明 | 常用度 |
|---|---|---|
| **read** | 离线只读：只能看，一个字都写不了，联网也关 | 偶尔 |
| **search** | 联网只读：能联网查资料/检索，但仍一个字都写不了 | 偶尔 |
| **work** | 能写，但每处改动按 L 级要你点头 | ★**日常主力** |
| **auto** | 无人值守时自动写，危险动作仍被拦 | 少 |
| **bypass** | 无视一切规则全放开——**唯一例外：读不了别人 Agent 的 S 秘密** | 极少（多半纸面） |

- read vs search 只差**联网与否**，其余相同（都硬 deny 写）。
- work 是绝大多数任务该用的档；auto 留给少见的无人长任务；bypass 基本只在纸面。

> **⚠ read 与 search 在 Claude 侧实现上无法区分**：两者都映射到 `plan` 模式，hook 收到的 `permission_mode` **只有 `"plan"` 一个值**，天生分不出是 read 还是 search。
> 因此 Claude 侧**统一按 search 处理**（联网只读放行）。"离线只读"在 Claude Code 里**没有对应的原生模式，做不到**。
> §三 原写的"read 额外禁 WebFetch/WebSearch/联网 Bash"是设计意图，**从未实现、也无法实现**，v0.0.3 起作废。

## 二、画像 → 各家原生设置（对照卡，切换时照点，不用背原生名）

| 画像 | 在 Claude 怎么进 | 在 Codex 怎么进 |
|---|---|---|
| read | `plan` 模式（+ 规则禁联网工具） | config：read-only + 无网 + approval never |
| search | `plan` 模式（联网工具放行） | config：read-only + 联网 + approval never |
| work | `manual`(default) 模式 | config：workspace-write + on-request |
| auto | `acceptEdits` 模式 | config：workspace-write + on-request（无人值守下越界无人批＝拦） |
| bypass | `bypassPermissions` 模式 | `--dangerously-bypass-approvals-and-sandbox` |

**切换画像**：对话中途可切、立即生效、**只有用户能切**——模型没有"改自己权限模式"的工具，连退出 plan 都要用户点头（已查官方文档证实）。代价：在 Claude UI 切换时会瞄到原生模式名（plan/manual/…），照上表点即可，不必懂原生名含义。

## 三、Claude 侧机制：画像 = permission_mode + cwd（直接读 payload，不用标签文件）

每次 PreToolUse hook 的输入 JSON 里**天生带** `permission_mode`、`cwd`、`session_id`（已查官方文档证实）。于是：

- **动作级别（哪档画像）= `permission_mode`**：每对话独立、用户设、**模型改不了**——这三条正是"画像开关"需要的安全属性。**不用** `.claude/画像.txt` 那种全工作区共享、还怕 AI 自己改的标签文件（早先的标签方案作废）。
- **项目范围 = `cwd`**：对话启动选的文件夹即范围，hook 从 payload 读。
- **read/search = `plan` 模式**：plan **原生禁掉一切文件写入、且模型无法自行退出**——"read/search 画像下一个字都写不了、要写先切画像"零成本天然实现。但 hook **分不出** read 和 search（payload 里都是 `plan`），故统一按 search 处理；原"read 额外禁 WebFetch/WebSearch/联网 Bash"一条**无法实现，已作废**（见 §一）。
- **work/auto/bypass** 分别 = manual / acceptEdits / bypassPermissions；写判定的细致收紧（L 级、范围）由 hook 在 manual/acceptEdits 下补。

⚠️ 部署到你的环境后建议实测三点（因 Claude Code 版本/环境而异，详见 §八）：① payload 是否带 `permission_mode`；② 子目录启动的对话是否仍加载根 `.claude/settings.json` 的 hook；③ **bypass 模式下 hook 是否仍触发**（决定 bypass 怎么落地，见 §五）。

## 四、文件写判定：L 级 × 画像 × 范围

> 本节**只管文件写**（Edit / Write / NotebookEdit）。Bash / WebFetch / WebSearch / MCP 的判定见 **§四之二**（v0.0.3 新增的画像射程）。

"范围内"＝目标在对话 cwd 子树内；"范围外"＝别的项目/更上层。**统一规则：范围外＝在范围内判定上"升一档严"**（allow→ask；无人可批的 auto → 范围外 deny）。

**范围内**（自己项目）：

| 画像 | L0 | L1 | L2 |
|---|---|---|---|
| read / search | deny | deny | deny |
| work | ask（+备份+tag） | ask | allow |
| auto | ask（无人应答即 deny） | allow | allow |
| bypass | 放行\* | 放行\* | 放行\* |

**范围外**（别的项目）：

| 画像 | L0 | L1 | L2 |
|---|---|---|---|
| read / search | deny | deny | deny |
| work | ask | ask | **ask**（较范围内 allow 升一档） |
| auto | deny | deny | deny |
| bypass | 放行\* | 放行\* | 放行\* |

- read/search：plan 原生禁写，范围内外、L 级全 deny。
- `*` **bypass 无视 L 级和范围，一律放行——唯一拦的是 S 读**（见 §五）。
- 根级 L0/L1 对"项目对话"永远算范围外，故总要 ask/deny，天然拦住"项目对话顺手改根级制度文件"。工作区之外的文件仍一律 exit 0 不管。

## 四之二、工具类判定：画像 × 工具类（v0.0.3 新增）

> **修复的问题**：此前 hook 只挂在 `Edit|Write|NotebookEdit` 上，**Bash / WebFetch / WebSearch / MCP 完全在画像管辖之外**，一律落回 Claude Code 原生白名单。而白名单**不继承父目录**（官方原文："Hooks **and other `settings.json` keys** load from the cwd's `.claude/` with **no parent-directory fallback**"）→ 子目录会话白名单归零 → **弹窗轰炸**：连 `echo`、空命令 `:`、`[ ! -s file ]` 都要弹窗批准，且大量调用来自 subagent（用户看不到上下文，只见孤零零的弹窗）。

**工具四分类**（按可逆性 + 危害面，不按名字）：

| 类 | 谁 | 特征 |
|---|---|---|
| **A 本地只读** | Read / Glob / Grep | 原生已放行，本来就不弹窗 |
| **B 联网只读** | WebFetch / WebSearch | 只读、不改本地 |
| **C 代码执行** | Bash、MCP 的 `exec` 档 | **危害无上限**；命令行文本判定不可靠 |
| **D MCP** | 其余 `mcp__*` | 按 `md/MCP工具分级表.md` 分三档 |

**判定矩阵**（与 §四 的文件写表并列，一起构成完整权限面）：

| 工具类 | read / search（plan） | work | auto | bypass |
|---|---|---|---|---|
| **A 本地只读** | allow | allow | allow | allow |
| **B 联网只读** | **allow** | **allow** | **allow** | allow |
| **C 代码执行** | 纯净只读 allow / 其余 **deny** | 纯净只读 allow / 其余 **divide** | 同 work | allow |
| **D MCP·只读** | **allow** | **allow** | **allow** | allow |
| **D MCP·写** | deny | **ask**（⚠ 白名单免疫不了） | **ask** | allow |
| **D MCP·代码执行** | deny | **divide** | **divide** | allow |
| **D MCP·未登记** | deny | **ask**（保守兜底） | **ask** | allow |

**C 类的天花板 = `divide`，永不 `allow`**：`python -` 的代码是从 stdin 喂进去的，**任何基于命令行文本的规则都拦不住它**——所以坚决不给代码执行类开 allow 的口子。唯一例外是下面这条。

**纯净只读命令**（hook 直接 allow，专治"被 `echo` 弹窗打断"的荒谬）：

- 命令头 ∈ `{echo, :, [, test, export, command, pwd, ls}`，或形如 `git status|log|diff|show`
- **且**整条命令**不含任何**拼接符 / 重定向 / 命令替换 / 换行：`|` `;` `&` `<` `>` 反引号 `$(` 回车
- 含任一拼接符 → **一律不放**（`echo x && rm -rf /` 也是 echo 开头）

**画像相对原生白名单的增量价值**（想清这条，设计才立得住）：原生白名单一旦点了"总是允许"就**永久放行、不管那操作多危险**。画像能 override 它，两端收口——read/search 下整类关死；**MCP 写类强制 `ask`，让白名单免疫不了不可逆操作**（哪怕手滑点过"总是允许"，删除仍每次问）；bypass 全放。中间那一大片 `divide` 交给白名单 + 人，**不跟原生抢活**。

**横切兜底**（不看画像，谁都拦）：Bash 命令触及 S 级路径 → **deny**（`check_permission.ps1` 自查 + `check_s_level.ps1` 双保险，不依赖多 hook 决策优先级）。

## 四之三、divide（分流）——第四种决策

hook 有四种决策，**语义完全不同，绝不可混用**：

| 决策 | 效果 |
|---|---|
| `allow` | **覆盖白名单**，直接执行，**永不弹窗** |
| `deny` | **覆盖白名单**，直接拒绝 |
| `ask` | **强制弹窗**——⚠ **连白名单里的也照问**，比 divide **更严**，绝不是"温和" |
| **`divide`（分流）** | hook **不表态**（`exit 0` 无输出）→ 落回 Claude Code 原生流程：**查白名单，命中即执行 / 未命中弹窗 ask**；work 的 ask 一直等人，auto 的 ask 无人应答即 deny |

**`divide` 既不是放行、也不是拒绝**——它是"把决定权**分流**给「白名单 + 人」"。这正是代码执行类该有的待遇：**不要去手搓一个命令危险性解析器**（那只会造出一个比原生更差、还自以为安全的东西），而是复用 Claude Code 原生那套声明式、可审计的白名单。

## 五、S/P 读判定：所有画像之下的"不下线地板"（含 bypass）

| 读取方 | P（公开） | S（秘密） |
|---|---|---|
| 拥有方 AI 自己 | allow | allow |
| 另一方 AI | allow | **deny（硬拦，任何画像，含 bypass）** |

- S 读禁**与画像无关**，从 read 一路到 bypass 都在——这是唯一从不放开的底线。
- 落地靠 `check_s_level` 这个 hook。**因此 bypass 的形态被这条倒推决定**：
  - 若实测 **bypass 仍跑 hook** → bypass 用原生 `bypassPermissions`，S 照拦；
  - 若实测 **bypass 关掉 hook** → bypass **不用原生 bypassPermissions**，改成"其它全放开、但保留 S 那个 hook"的配置。
- 一句话：**别的都能放开，别人的 S 秘密永远读不了。**

## 六、auto 画像下的 L0（走 ask，不用令牌）

auto（无人值守）下 L0 走 `ask`：人在（哪怕在别的对话）可确认放行；无人应答（无头/长时间无人）则 deny——既不卡合法变更、也不静默放行、更不永远阻塞。原一次性令牌机制已弃用：交互场景切 manual 确认即可，令牌那条无人值守旁路基本用不上。

## 七、必须知晓的事实约束

- **Windows 无 OS 沙箱**：work-pc 原生 Windows 无 sandbox（仅 mac/Linux/WSL2 有），写保护全靠 hook + 权限规则，应用级非 OS 级；`denyRead` 等沙箱特性不可用。
- **文件层强制不对称**：Claude 编辑前拦（deny→文件没改）；Codex git pre-commit 提交前拦（文件已改、进不了 git）。二者不等价。
- **ask 语义**：deny 任何模式硬拦；ask 交互会话弹窗等人、真无头才穿透。
- **三个"自动/少问"模式的实测区别**（决定映射取舍）：

| 模式 | 不问就放行 | 仍拦/仍问 |
|---|---|---|
| acceptEdits | 读、编辑、7 个安全文件命令（**限工作目录**） | 工作目录外、**受保护路径 `.git/` `.claude/`**、其它 Bash、Web/Agent |
| auto（原生） | 一切（后台分类器审危险） | 分类器命中的危险操作 |
| dontAsk | 仅 allow 规则 + 只读命令 | 其它一律**直接拒**（不弹窗）——"白名单外全拒"的无头模式 |

  → **work→auto 画像绑 `acceptEdits`**（它原生还替我们对 `.claude/` 受保护路径把关＝L0 的一道原生兜底）；原生 `auto` 模式更激进、留作可选；**`dontAsk` 不属任何画像**（限制型无头模式，不是"自动写"）。

> 注意：表中"auto（原生）"是 Claude 的一个**原生权限模式**名，与我们的 **auto 画像**不是一回事——我们的 auto 画像默认绑的是 `acceptEdits` 模式。

## 八、部署到你的实际工作区后的验证建议

把本套 hook/配置装进你自己的 Claude 工作区后，建议实测以下几点（结果因 Claude Code 版本/环境而异）：

1. **hook 能读到 `permission_mode`**：在 hook 里打一条日志看 payload——本套画像判定依赖它。
2. **子目录启动是否加载根 hook**：在子目录启动的对话，是否仍加载工作区根 `.claude/settings.json` 的 hook。
3. **bypass 下 hook 是否仍触发**：决定 `bypass` 画像的落地形态（见 §五）。
4. **S 读隔离生效**：在 `md/S级清单.md` 填一个测试路径，用 Read/Bash 访问，确认被 deny。

> 注意：改 `.claude/settings.json` 属 L0；hook 在会话启动时加载，改完需**新开会话**才生效。
