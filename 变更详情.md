3. **审批记录入库风险—不成立**：审计日志仅含 action/file/level/commit，无敏感字段；`~/.codex/` 配置在家目录，不受 git 跟踪。

## v0.0.4（2026-07-15）

### 每日工作总结漏采子目录 cwd 会话（bug）
- **问题**：每日工作总结自动化的抽摘要步骤 `Automation/extract_day.py` 只采集根工作区 cwd 对应的那一个 Claude project 目录；子目录（子项目）作为独立 cwd 启动的会话存在各自的 project 目录（`~/.claude/projects/<根编码>-<子路径编码>`），完全不被采集 → 每日总结漏掉该子项目当天工作。"文件变更记录"因基于 git 全覆盖没漏，两者不对称正好暴露此缺陷。
- **根因**：`find_projects_dir(wsroot)` 三级定位（env 覆盖 / `encode_path` 精确 / 末段唯一命中）三条都只解析到**根那一个**目录，子目录 cwd 的目录（以 `<根编码>-` 为前缀）一个都不在返回值里。
- **修复（A 治本）**：`find_projects_dir` → `find_projects_dirs`——先定位根 project 目录名，再 `glob(<根名>)` + `glob(<根名>-*)` 纳入根 cwd + 所有子目录 cwd 会话；`main()` 遍历多目录合并（uuid 去重已有，天然防跨目录/续聊重复）。**不硬编码任何工作区名**（用 `encode_path`/fallback 通用推导根名），保持模板通用性。实测：本地路径喂入返回 3 个 project 目录（根 + 两个子目录），端到端抽取 user 34→61。
- **修复（B 兜底）**：`Automation/daily_worklog.sh` 的 `COMMON_RULES` 加第 6 条"文件变更兜底"铁律——凡某项目/子目录当天有实质文件产出（源自 git、全覆盖）却在对话摘要里没有的，据文件变更补写一句。即使 A 哪天再漏新会话，git 文件变更是最后一道防线。
- **涉及**：`Claude/Automation/extract_day.py`、`Claude/Automation/daily_worklog.sh`。

### Codex 侧同病（Codex 已修复 2026-07-15）
- 核实 `Codex/Automation/extract_day.py`：`glob(sessions/**/*.jsonl, recursive=True)` 扫全部文件（文件层面不漏），**但 `matched` 过滤用 `norm_path(session_cwd(f)) == target_cwd` 精确匹配根 cwd**，子目录 cwd 会话（`cwd = <根>/<子目录>`）被过滤掉 → 等价盲区（病灶在"cwd 精确匹配"，非"文件没扫到"）。
- 建议修法：`== target_cwd` 改子树匹配 `c == target_cwd or c.startswith(target_cwd + os.sep)`。
- **交 Codex 认领修改**；详见 `多AI交流_本地工作改造.md §五`。


### check_s_level.ps1 大重写（安全增强，重大）

在原有 S 级读取拦截基础上，新增三层拦截，并修复多个 Windows 兼容性 bug：

- **新增：L0 文件删除拦截（deny）**— 命令含 rm/del/Remove-Item 等删除操作且目标是 L0 文件 → 硬拦（需双确认 + 安全审核.ps1 -Approve）
- **新增：L1 文件删除拦截（ask）**— 同上但目标是 L1 文件 → 请求批准（交互模式弹窗）
- **新增：外部路径写入/删除拦截（ask）**— 命令涉及工作区外的绝对路径且不是纯读操作 → 请求批准。读操作放行。
- **修复：paths 围栏提取从正则改为 IndexOf**— PowerShell 子进程中正则的反引号转义问题导致 S 级拦截失效，改用 IndexOf 字符串操作避免。
- **修复：围栏字符从 `[char]96 * 3` 改为 `[string]::new([char]96, 3)`**— PowerShell 不支持 char 乘法。
- **修复：文件读取从 Get-Content 改为 `[System.IO.File]::ReadAllText`**— Get-Content 在子进程中编码处理有问题，ReadAllText 正确处理 UTF-8 BOM。
- **修复：外部路径正则加入 winFwdPaths**— 原仅匹配反斜杠路径，现加入正斜杠。
- **测试**：S 级读取 deny ✅ | L0 删除 deny ✅ | L1 删除 ask ✅ | 外部写入 ask ✅ | 普通读取放行 ✅

### after_turn.ps1 增强（安全增强，重大）

在原有回合结束审计基础上，新增删除检测 + 审批检查 + 自动恢复：

- **新增：删除检测**— 回合结束后 git status 检查 D 状态的 tracked 文件，即使经 os.remove / 提权脚本等绕过运行时拦截，Stop hook 仍能检测到。
- **新增：审批检查**— 恢复前读取审计日志，查找该文件是否有活跃审批（APPROVE 后无 LOG）。有审批 → 只记录不恢复；无审批 → 记录 + 恢复 + 告警。
- **新增：自动恢复**— 无审批的删除 → git checkout 自动恢复 + stderr 告警。
- **修复：审批检查误判**— file-deleted-detected 条目被当作 LOG 导致活跃审批被误判为已使用，修复为仅 action=LOG 才标记审批已用。
- **测试**：无审批删除恢复 ✅ | 有审批删除跳过恢复 ✅

### 解决的系统性缺口（P012）

P012（提权命令绕过 hook 删除文件）现有两层防线：
1. PreToolUse hook（check_s_level.ps1）— 拦截 Bash 命令中的 L0/L1 删除和外部写入，返回 deny/ask。
2. Stop hook（after_turn.ps1）— 卷底：回合结束后检测并恢复被删除的 tracked 文件，无论如何删除。

## v0.0.3（2026-07-14）

### 画像射程扩张 · 从"只管文件写"到"管全部四类工具"（重大）
- **问题**：`check_permission.ps1`（画像判定）此前只挂在 `Edit|Write|NotebookEdit` 三个工具上——**Bash / WebFetch / WebSearch / MCP 完全在画像管辖之外**，全部落回 Claude Code 原生白名单。而白名单和 hooks 一样**不继承父目录**（与 v0.0.2 D2 同一根因；官方原文 "Hooks **and other `settings.json` keys** load from the cwd's `.claude/` with **no parent-directory fallback**"——v0.0.2 只读走了 hooks 那半句，permissions 这半句在同一行）。
- **后果**：以子文件夹为 cwd 的会话白名单归零 → 每个 Bash 命令 / 每个网页域名 / 每个 MCP 工具都要重新弹窗批准。实测一个检索会话有 185 次需审批调用（Bash 61 / WebFetch 63 / WebSearch 48 / MCP 13），其中约 149 次来自 subagent（用户看不到 subagent 上下文，只见孤零零的弹窗）。连 `echo`、空命令 `:`、`[ ! -s file ]`、`export` 都在弹窗。
- **为什么不靠"铺白名单"修**：Bash 命令是无穷组合（每个新 URL、新 echo 都是新规则），白名单永远补不完。根治办法是让画像**接管** Bash / Web / MCP。
- **落地**：`check_permission.ps1` 入口改为按 `tool_name` 分流，实现「画像 × 工具类」矩阵：
  - **A 本地只读**（Read/Glob/Grep）：原生已放行，不介入。
  - **B 联网只读**（WebFetch/WebSearch）：一律 `allow`（只读、不改本地，联网查资料是天职）。
  - **C 代码执行**（Bash、MCP 的 exec 档）：天花板 = `divide`，**永不 allow**（`python -` 的代码走 stdin，命令行文本规则拦不住）；唯一例外"纯净只读命令"直放（命令头 ∈ `{echo,:,[,test,export,command,pwd,ls}` 或 `git status|log|diff|show`，且不含 `| ; & < > 反引号 $( 回车` 任一拼接符）。
  - **D MCP**：按新增的 `md/MCP工具分级表.md` 分三档——readonly→allow / write→ask（白名单免疫不了）/ exec→同 C；表里没有的按工具名模式兜底，**未登记默认 ask**。
- **横切兜底**：Bash 命令触及 S 级路径 → `deny`（`check_permission` 自查 + `check_s_level` 双保险，不依赖多 hook 决策优先级）。
- **涉及**：`hooks/check_permission.ps1`（分流 + 矩阵 + S 自查）、`.claude/settings.json`（matcher 扩到 `...|Bash|WebFetch|WebSearch|mcp__.*`）、三脚本 L1 清单加 `md\MCP工具分级表.md`、`md/权限系统.md`（§二·补二）、`md/画像映射表.md`（§四之二/四之三）、`CLAUDE.md`（§四）。

### 新术语：divide（分流）
- hook 有四种决策：`allow`（覆盖白名单直接放）/ `deny`（覆盖白名单直接拒）/ `ask`（**强制弹窗，连白名单里的也照问**）/ **`divide`**（不表态、`exit 0` 无输出 → 落回原生白名单：命中即执行、未命中弹窗 ask；work 一直等、auto 无人应答即 deny）。
- `divide` 既不是放行也不是拒绝，是"把决定权分流给「白名单 + 人」"。用于代码执行类——不自己手搓命令危险性解析器，复用 Claude Code 原生那套声明式、可审计的白名单。

### 新文件：md/MCP工具分级表.md（功能 / 新增）
- L1，被 `check_permission.ps1` 直接读取（三个围栏 `readonly`/`write`/`exec`）。
- **模板内为空壳**（三个围栏无条目 + 示例注释）：MCP 工具名是各部署环境特有的（zotero / playwright / matlab…），不塞进公开模板。部署方按自己接入的 MCP 填写；不填也能跑（走模式兜底 + 未登记 ask）。

### 安全修复：.gitignore 补 settings.local.json（重要）
- 模板 `.gitignore` 原本**完全没有** `settings.local.json` 的排除规则。而该文件是用户点"总是允许"累积的白名单，含**本机绝对路径、邮箱、常访问域名**（= 个人画像），属敏感信息。
- 补 `**/.claude/settings.local.json`（及 sessions/cache/logs 等）。**必须带 `**/` 前缀**：自"钩子覆盖件"起每个子目录都有自己的 `.claude/`、各自生成一份 local，不带 `**` 只匹配根级那一份、子目录全漏。

### 勘误：read 与 search 无法区分
- 两者在 Claude 侧都映射到 `plan` 模式，hook 拿到的 `permission_mode` 只有一个值，天生分不出。实际统一按 search 处理（联网只读放行）。原设计"read 额外禁 WebFetch/WebSearch/联网 Bash"从未实现、也无法实现，作废。

## v0.0.2（2026-07-14）

### D2 · 子文件夹会话不触发根钩子（bug，重大）
- **问题**：Claude Code 只从"对话工作目录自身的 `.claude`"加载 hooks、**不回溯父目录**（官方设计，无继承开关）。以子文件夹为 cwd 开的会话不加载根钩子 → 绕过门禁 + 审计 + S 隔离。
- **实证**：分别在"自带 `.claude` 的子目录"和"不自带的子目录"起会话做探针，均确认不触发；修复后同样探针触发（自动提交 + 审计）。
- **修复**：每个可能被当 cwd 打开的目录各放一份 `.claude/settings.json`（注册根 `hooks/` 三脚本，绝对路径，脚本自行算工作区根）；`check_permission.ps1` / `after_edit.ps1` / `安全审核.ps1` 三脚本加 `*\.claude\settings.json → L1`。
- **部署注意**：模板不含具体子项目的覆盖件——部署方须按自己的目录树**逐层铺**（复制根 `.claude/settings.json` 即可）。详见 `Claude/md/权限系统.md §3.6`。

### 去令牌 → auto + L0 = ask（功能 / 设计变更）
- **变更**：auto 画像原对 L0 用"一次性令牌"（预放令牌文件才放行一次并消费）；改为 `ask`——人在（哪怕在别的对话）可确认放行，无人应答（无头 / 长时间无人）则 deny；work 的 L0 则一直等、不自动 deny。
- **理由**：交互场景有人在就能确认，令牌那条无人值守旁路基本用不上；去掉更简单、也少一个要管的令牌文件。
- **涉及**：`hooks/check_permission.ps1`（删令牌块、L0→ask）、`md/画像映射表.md`（§四表 + §六）、`md/权限系统.md`（§二·补）、`md/hooks策略.md`、`CLAUDE.md`。

### 权限文档改名（维护）
- `md/文件权限系统.md` → `md/权限系统.md`，含脚本 `$L0_files`、`CLAUDE.md`、`test-agent` / `design-agent`、`hooks策略.md` 及文档标题等全部引用。

> **仅本地、不入本模板的一处**：本地工作区还把"权限清单三处→两处"做了订正（因本地 `安全审核.ps1` 已简化成纯查询、不持清单）。**模板不适用**——模板的 `安全审核.ps1` 是支持无 hook Agent 的完整版、确实持清单，"三处"在模板里本就正确。

### git-hooks shebang 修复（bug，Windows 兼容）

- **问题**：Codex 侧 `git-hooks/pre-commit` 和 `post-commit` 使用 `#!/bin/bash` shebang。在 Windows 上从 PowerShell 调用 git 时，git 无法找到 `/bin/bash` 解释器（报 `cannot spawn ...: No such file or directory`），导致 L0/L1 提交门禁和审计日志自动写入完全失效。
- **修复**：`#!/bin/bash` → `#!/bin/sh`。Git for Windows 自带 `sh` 解释器，能正确找到并执行。
- **测试**：L0 提交被阻止、L1 告警放行、L2 静默通过、post-commit 实计日志自动写入、`--no-verify` 可绕过 pre-commit 但 post-commit 仍执行。全部通过。
- **影响范围**：`Codex/git-hooks/pre-commit` + `Codex/git-hooks/post-commit`。Claude 侧无 git-hooks（用 PostToolUse hooks 替代），不受影响。

### after_turn.ps1 自动提交逻辑（新增功能）

- **问题**：原 after_turn.ps1 只写审计日志 + 检测删除文件，**不做 git commit**。导致 AI 会话中修改的文件不会自动提交到 git，后续修改可能覆盖前面的内容且无法回退。
- **修复**：新增第 3 段自动提交逻辑：回合结束时 `git add -u` + `git commit --no-verify`（跳过 pre-commit，post-commit 仍执行）。只提交已跟踪文件变更，不提交未跟踪文件（`??` 状态排除）。
- **关键细节**：用 `git.exe` 全路径（不依赖 PATH）、等 500ms 让磁盘写入刷新、调试日志写到 `.after_turn_debug.log`。
- **影响范围**：`Codex/hooks/after_turn.ps1`。

### hooks.json 格式修复（bug）

- **问题**：hooks.json 原用扁平格式（PreToolUse 等在顶层），Codex CLI 要求 `hooks` 包装格式（`{"hooks": {"PreToolUse":...}}`）。格式错误导致 hooks 不执行。
- **修复**：改为 `hooks` 包装格式。同步更新 `hooks.json.模板`。

### after_turn.ps1 正则 bug 修复（bug）

- **问题**：路径检测用的正则 `^??` 中 `?` 是正则量词（“零或一个空”），匹配所有字符串，导致 `-notmatch "^??"` 永远为 False，auto-commit 永远不触发。
- **修复**：`^??` → `^\?\?`（转义问号为字面字符）。

### after_turn.ps1 中文路径 bug 修复（bug）

- **问题**：脚本生成时中文路径 `审计日志.jsonl` 被写成了字面的 `\u5ba1\u8ba1...`，导致 `Add-Content` 找不到文件。
- **修复**：将 `\u` 转义序列替换为实际 UTF-8 中文字符。

### hook trust 机制说明（文档补充）

- Codex CLI 有 hook trust 安全机制：hooks 配置后默认不执行，需用户在 Codex 设置中手动信任。详见 `hooks策略.md`。
- `codex_hooks = true` 不是有效的 config.toml 设置（已删除）。hooks 功能默认已启用（`stable + true`），无需手动开启。
