# 文件权限系统（L0 级）

> 本文件定义工作区中所有文件的权限、审批、审计和多终端修改规则。
> AI 每次会话开始时应首先读取根级 `README.md`，仅在权限制度变化时再读本文件。
> 文件路径以当前工作区根目录为准。

---

## 一、基本原则

1. 工作区允许通过 git 在多台终端间同步文件，但所有修改必须可追溯。
2. 默认使用中文交流和生成中文文档。例外：代码、命令、API 名称、标准英文文件名、英文项目目录名可保留英文。
3. 所有路径说明优先使用"工作区根目录"表达，避免硬编码某一台终端的绝对路径。
4. `审计日志.jsonl` 只追加，不修改历史；面向人查询用 `安全审核.ps1`。
5. **审计写入由 hooks 自动完成**，不依赖 AI 自觉调用 `安全审核.ps1`。

---

## 二、文件权限分级

### 2.1 文件内容修改权限

| 级别 | 名称 | 修改规则 |
|:--:|------|------|
| **L0** | 权限系统 | 需两次用户确认 + 自动备份 + git tag + 审计 |
| **L1** | 指令锁 | 需用户批准 + 审计 |
| **L2** | 正常 | 开发流程中自由修改，hooks 自动审计 |

### 2.2 文件操作权限（新增 / 删除 / 移动）

| 操作 | 级别 | 规则 |
|------|:--:|------|
| 新增文件 | L1 | 需用户批准 + 审计 |
| 删除文件 | L0 | 需两次用户确认（不终止对话） + 自动备份 + git tag |
| 移动/重命名文件 | L0 | 需两次用户确认 + 自动备份 + git tag |

> 删除和移动为 L0 级别，但仅需对话内双次确认，无需终止或重启会话。

### 2.3 生命周期升降级

- 版本开发阶段：该版本的需求/设计/入口文件为 L2
- 版本标记为"已完成"后：自动升级为 L1
- 版本标记为"迭代中"后：降回 L2

## 四、自动化操作流程（git hooks + 手动批准流）

> Codex 的 PreToolUse/PostToolUse **仅匹配 Bash**，不拦截 Edit/Write/Read。
> 文件编辑的权限检查和自动审计改用 **git hooks（pre-commit/post-commit）+ `安全审核.ps1` 手动批准流**。

### 4.1 L2 修改流程（最快，git hooks 全自动）

```text
① AI 修改文件
② AI 执行 git add <文件> && git commit
③ pre-commit hook 检查文件级别 → L2，直接放行
④ post-commit hook 自动触发：
   ├─ 追加一条 JSON 到 审计日志.jsonl
   └─ 记录 commit hash、文件、级别
```

### 4.2 L1 修改流程（一次确认）

```text
① 安全审核.ps1 -Request -File "路径" [-Reason "原因"]  → 记 REQUEST
② 用户批准：安全审核.ps1 -Approve -File "路径"        → 记 APPROVE
③ AI 修改文件
④ 安全审核.ps1 -Log -File "路径" -Summary "摘要"       → 记 MODIFY
⑤ AI 执行 git add <文件> && git commit
⑥ pre-commit hook 检查 L1 → 已批准则放行，未批准则告警
⑦ post-commit hook 自动写审计
```

### 4.3 L0 修改流程（两次确认 + 加固）

```text
① 安全审核.ps1 -Request -File "路径" [-Reason "原因"]
   ├─ 自动复制原文件到 .backups/L0/{文件名}.{时间戳}.bak
   └─ git tag pre-L0-{时间戳}-{文件简称}
② 第一次用户确认
③ 用户批准：安全审核.ps1 -Approve -File "路径"
④ 第二次独立确认
⑤ AI 修改文件
⑥ 安全审核.ps1 -Log -File "路径" -Summary "摘要"
⑦ AI 执行 git add <文件> && git commit
⑧ pre-commit hook 检查 L0 → 已批准则放行，未批准则阻止（exit 1）
⑨ post-commit hook 自动写审计（commit message 含 [L0] 标记）
```

### 4.4 批准指令

用户输入以下任一指令视为批准：`yes` / `同意` / `是` / `y` / `Y`
以下指令视为拒绝：`no` / `拒绝` / `否` / `n` / `N`

### 4.5 用户手动修改文件

用户绕过 AI 直接编辑文件时，**Codex 原生 hooks 不会触发**（PreToolUse/PostToolUse 只匹配 Bash）。但用户提交时 **git hooks（pre-commit/post-commit）对所有提交生效**——无论是 AI 还是人类提交，都会经过 pre-commit 的 L0/L1 检查和 post-commit 的审计日志写入。

---

### 4.6 手动批准流（安全审核.ps1）

L0/L1 文件的修改使用 `安全审核.ps1` 的手动批准流：

```text
① 安全审核.ps1 -Request -File "路径" [-Reason "原因"]   L0 自动备份+打 tag，记 REQUEST
② 用户：安全审核.ps1 -Approve -File "路径"              记 APPROVE
③ AI 修改文件
④ 安全审核.ps1 -Log -File "路径" -Summary "摘要"        记 MODIFY
```

git hooks（pre-commit/post-commit）和 `安全审核.ps1` 写同一份 `审计日志.jsonl`（schema 一致），查询统一用 `-Report`。

## 五、安全审核脚本（统一：查询 + 手动批准流）

`安全审核.ps1` 兼顾查询与手动批准流（git hooks 在提交时自动完成审计，手动批准流见 §4.6）：

| 命令 | 说明 |
|------|------|
| `-Report` | 显示全部审计记录 |
| `-Last <N>` | 显示最近 N 条 |
| `-File <文件名>` | 过滤：某文件的全部修改 |
| `-Request -File "路径" [-Reason "原因"]` | 发起审批请求（L0 自动备份+打 tag） |
| `-Approve -File "路径"` | 批准审批请求 |
| `-Deny -File "路径"` | 拒绝审批请求 |
| `-Log -File "路径" -Summary "摘要"` | 记录文件修改完成 |
| `-Level <级别>` | 指定/查询文件级别 |
| `-Help` | 显示用法 |

## 六、多终端修改规则

1. 不同终端通过 git 同步；冲突时以**语义正确**为准，不盲信任一方时间戳。
2. 审计日志按时间线追加，不因终端不同而分段；commit message 的 `[审计ID]` 是跨终端追踪锚点。
3. 终端硬件/账号/代理环境如需登记，使用者可在 `md/` 自建登记文档（本框架默认单终端，不强制）。

---

## 七、维护规则

1. 权限制度变化 → 更新 §二、§三，同时同步 `安全审核.ps1`、`git-hooks/pre-commit` 与 `git-hooks/post-commit` 的权限清单数组。
2. 本文件 L0，修改需两次用户确认 + 自动备份 + git tag（由 hooks 自动加固）。

## 八、阅读权限分级（S/P 等级）

> S/P 是**阅读维度**的权限，与 L0/L1/L2（修改维度）正交。
> L0/L1/L2 管"能不能改、改了要什么审批"；S/P 管"能不能读"。

### 8.1 等级定义

| 等级 | 含义 | 行为 |
|---|---|---|
| **S（Secret 秘密级）** | 涉密材料 | 拥有方 AI 可读可改；**其他 AI 不可读**（触发时直接 deny，无需人工介入） |
| **P（Public 公开级）** | 一般文件 | 所有 AI 都可自由阅读 |

### 8.2 归属规则

- S 级文件**属于各自工作区**：Codex 工作区的 S 级文件，Codex（拥有方）可读写，Claude 不可读；反之亦然。
- 一般文件默认为 P 级。
- 只有涉密材料（如公司标准、商业机密）才设为 S 级。

### 8.3 .gitignore 规则（必须）

**所有 S 级文件/目录必须列入 `.gitignore`，确保涉密材料永不提交到 git。**

每个工作区的 `.gitignore` 必须包含一个 `S 级文件` 段落，列出该工作区内所有 S 级路径。示例：

```gitignore
# === S 级文件（秘密级，不可入库）===
# 所有标记为 S 级的文件/目录必须在此列出
confidential/
```

### 8.4 执行机制

**check_s_level.ps1**（PreToolUse hook）：拦截 Read（Claude）和 Bash（Claude + Codex）工具中对 S 级路径的访问。

| AI | Read 工具拦截 | Bash 命令拦截 | OS 级读禁 | 配置位置 |
|---|---|---|---|---|
| **Claude** | PreToolUse 匹配 Read → deny ✅ | PreToolUse 匹配 Bash → check_s_level.ps1 ✅ | sandbox.filesystem.denyRead（仅 macOS/Linux） | `.claude/settings.json` |
| **Codex** | 无 Read 工具 | PreToolUse 匹配 Bash → check_s_level.ps1 ✅ | **不支持**（allow-list 模型，无 deny 条目） | `.codex/config.toml + hooks.json` |

**hook 脚本工作流程**：
1. 从 `md/S级清单.md` 的 `` ```paths `` 代码块读取对方 AI 的 S 级路径
2. 归一化为绝对路径，生成正斜杠/反斜杠/双反斜杠三种变体
3. 检查命令/路径是否包含任何 S 级路径变体
4. 命中 → `permissionDecision: "deny"`（硬拦，无需人工）
5. 未命中 → exit 0（放行）

**已知盲区**：
- 自动注入上下文（CLAUDE.md / memory / SessionStart）不是 Read/Bash 调用 → 拦不到。**约定**：S 级文件不得放入会被自动加载的位置。
- Codex sandbox 不支持 per-path denyRead → 只能靠 hook + .gitignore + AGENTS.md 约定。

### 8.5 S 级清单管理

- S 级路径在各自工作区的 `md/S级清单.md` 的 `` ```paths `` 代码块中维护（hook 从中读取）。
- S 级文件同时列入 `.gitignore`（保证不提交），但 `.gitignore` 还含日志/临时等非 S 项，两者分开维护、保持同步。
- 变更 S 级标记时，同步更新 `md/S级清单.md` 和 `.gitignore`。

### 8.6 落地状态

S/P 阅读等级已定稿并实现。跨 AI 读取拦截方案见 §8.4。
