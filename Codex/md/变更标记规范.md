# 变更标记规范

> **文件级别**：L1
> **维护者**：系统Agent + 用户批准
> **目的**：规定如何在文档里标记"上一次的变更"，让人和 AI 在 Obsidian 中**一眼看出最近改了什么**。
> **适用读者**：所有编辑工作区文档的人和 AI（含各 subagent）。

---

## 一、为什么需要这套规范

文件改完后，人和 AI 都不容易看出"相对上一次，哪些内容变了"。git diff 精确但面向机器、需要工具、不直观。本规范在**正文里直接用颜色标记最近一次的变更**，作为"快速一眼识别"的补充层。

**与 git 的分工**：

| | git diff / log | 本规范的正文标记 |
|---|---|---|
| 面向 | 机器 / 精确追溯 | 人 / 快速一眼识别 |
| 范围 | 全部历史 | **只标最近一次** |
| 载体 | 需要 git 工具 | 直接在 Obsidian 正文里看 |

**两者互补**：完整历史永远靠 git；正文标记只服务"最近一次改了啥"。

---

## 二、核心原则（务必遵守）

1. **正文只保留"最近一次"的标记**。下一轮编辑前，先清掉上一轮的旧标记（见 §六），再给本轮改动打标记。
2. **不在正文累积多轮标记、不做历史表**。要看演化用 `git log`。
3. **HTML 只用于"短的行内片段"，绝不用 HTML 包大段或代码块**。大块一律用 Obsidian callout（§四之 C）。
4. **颜色克制在 3 色**（金/绿/红），不滥用。

---

## 三、颜色 / 语义图例

| 语义 | 颜色 | 十六进制 | emoji |
|---|---|---|---|
| 修改（内容改动） | 金 DarkGoldenrod | `#B8860B` | 🟡 |
| 新增（新内容） | 绿 ForestGreen | `#228B22` | 🟢 |
| 删除（移除内容） | 红 Crimson | `#C0392B` | 🔴 |

---

## 四、三种标记的具体写法

### A. 行内短语 / 标签 / 表格单元格 → `<span>`

只用于**短**片段（一个词、一句话、一个表格格），不含代码块：

```markdown
这是 <span style="color:#228B22">新增的一句话</span>。
某参数从 100 <span style="color:#B8860B">改为 200</span>。
| 字段 | <span style="color:#228B22">新增列</span> |
```

### B. 行内代码符号 → `<code>`

由于使用 html 语法，特殊符号和希腊字母需转义为 Unicode 编码：

```markdown
这是 <span style="color:#228B22">新增的符号<code>&#948</code>一句话</span>。
```

### C. 整段 / 含代码块 / 含列表的大块 → Obsidian callout

**绝不用 HTML 包大块**。用 callout，每行加 `>` 前缀，内部仍是正常 markdown：

```markdown
> [!success] 🟢 新增：xxx 流程
> 这里可以放正常段落、列表、甚至代码块：
> ```bash
> echo "代码块在 callout 里也正常"
> ```
> - 列表项也正常
```

callout 类型与颜色对应（Obsidian 原生，无需 CSS）：

| 语义 | callout 写法 | 原生颜色 |
|---|---|---|
| 新增大块 | `> [!success] 🟢 新增：…` | 绿 |
| 修改大块 | `> [!warning] 🟡 修改：…` | 黄 |
| 删除大块 | `> [!danger] 🔴 删除：…` | 红 |

### D. 删除的内容 → 删除线 + 红

被删的文字若要让人看到"删了什么"，用原生删除线包红 span：

```markdown
<span style="color:#C0392B">~~这段被删除了~~</span>
```

> 注：纯删除（不需要让人看到旧内容）可直接删掉、不留标记——是否保留删除痕迹由编辑者按可读性判断。

## 五、标记的生命周期

```text
文件当前状态（带上一轮标记）
        │
        │  AI 准备再次编辑此文件
        ▼
① 先清除上一轮所有标记（§六）         ← 正文恢复干净
        │
        ▼
② 做本轮实际编辑
        │
        ▼
③ 给本轮改动打上标记（§四）           ← 正文只剩本轮标记
        │
        ▼
保存（hook 自动 commit）
```

- **谁加标记**：AI 编辑文件时同步加（hook 做不了，因为它不知道改动的语义意图）。
- **谁清旧标记**：AI 在**下一次编辑该文件前**自动清。
- **历史**：永远靠 git，不在正文累积。

---

## 六、清除上一轮标记的操作

编辑某文件前，先全文检索这些标志并按类处理：

| 标志（搜索目标） | 类型 | 清除动作 |
|---|---|---|
| `<span style="color:#228B22">X</span>` | 行内新增 | 去掉 span 标签，保留 `X` |
| `<span style="color:#B8860B">X</span>` | 行内修改 | 去掉 span 标签，保留 `X` |
| `<span style="color:#C0392B">~~X~~</span>` | 行内删除 | 整体移除（`X` 已是被删内容） |
| `> [!success] 🟢 新增：…` 块 | 新增大块 | 去掉 callout 包裹（每行 `> ` 前缀 + 标题行），保留内部内容 |
| `> [!warning] 🟡 修改：…` 块 | 修改大块 | 同上，去包裹保留内容 |
| `> [!danger] 🔴 删除：…` 块 | 删除大块 | 整块移除 |

**检索锚点**（grep 这几个就能找全）：`color:#228B22`、`color:#B8860B`、`color:#C0392B`、`[!success] 🟢`、`[!warning] 🟡`、`[!danger] 🔴`。

---

## 七、适用范围

| 类别 | 是否标记 | 说明 |
|---|---|---|
| 治理 / 需求 / 设计 / 长说明文档（AGENTS.md、md/ 各制度文件、各设计稿） | ✅ 标记 | 人会反复读，需要知道又改了啥 |
| 审计日志（审计日志.jsonl） | ❌ 不标记 | 本就是追加式时间线 |
| 代码 / 脚本（.ps1 .sh 等） | ❌ 不标记 | 用 git diff 看；标记会污染代码 |
| 模板文件 | ❌ 不标记 | 模板本身是稳定骨架 |

---

## 八、AI 操作清单（编辑文档时）

- [ ] 1. 编辑前：grep §六 的锚点，**清掉上一轮标记**
- [ ] 2. 做本轮实际内容修改
- [ ] 3. 给本轮改动按 §四 打标记（短语用 span / 大块用 callout / 删除用删除线）
- [ ] 4. 自检：正文是否只剩本轮标记、有没有用 HTML 包大块（违规）
- [ ] 5. 保存（hook 自动审计 + commit）

## 九、维护本文件

| 触发 | 动作 |
|---|---|
| 颜色 / 语义调整 | 更新 §三、§四 |
| 清除规则变化 | 更新 §六 |
| 适用范围变化 | 更新 §七 |

本文件 L1，修改需用户批准 + 审计（hooks 自动）。
