# Codex 画像映射表

> 画像是 agent-workspace 的共享抽象层，命名描述能力、不模仿原生名。
> 两家 AI 共用同一套用户词汇（read / search / work / auto / bypass），原生配置名降级为"对照卡"。
> 本文件定义每个画像在 Codex 原生配置中的对应值。

---

## 画像总览

| 画像 | 含义 | UI 等价 |
|---|---|---|
| **read** | 离线只读，一个字写不了 | Custom |
| **search** | 联网只读，能查不能写 | Custom |
| **work** | 受控写，日常主力 | Ask for approval |
| **auto** | 无人自动写，危险仍拦 | Ask for approval（无人应答→越界拦） |
| **bypass** | 无视规则全放开，唯留 S 读禁 | Full access |

旧名对照：read←lock, search←view, work←dev, auto←auto, bypass←(新增)

## 画像 → Codex 原生模式映射

| 画像 | sandbox_mode | approval_policy | network | 场景 |
|---|---|---|---|---|
| **read** | read-only | never† | false | 陌生环境、离线只读 |
| **search** | read-only | never† | true | 联网查资料、只读分析 |
| **work** | workspace-write | on-request | true | 项目内日常开发 |
| **auto** | workspace-write | on-request | true | 项目内长期任务、无人值守 |
| **bypass** | danger-full-access | never | true | 全放开（极少使用） |

> † `never` + `read-only`：Codex 回退到 "read-only + approvals disabled" = 真只读（写不了、也不给提权）。这是用户选择的预期行为，非错误。
> auto 与 work 的 config.toml 值相同，区别在使用场景——work 有人应答审批请求，auto 无人应答→越界请求无人批=拦截。

## Codex 原生配置项说明

### approval_policy（审批策略）

| 值 | 二进制文档原文 | 含义 |
|---|---|---|
| `untrusted` | "only 'known safe' commands that only read files are auto-approved. Everything else will ask." | 最严格交互：只有已知安全的纯只读命令免审批，其余（含写）一律弹窗 |
| `on-request` | "The model decides when to ask the user for approval." | 标准交互：沙箱范围内直接执行，越界弹窗请求用户批准 |
| `granular` | "Fine-grained controls for individual approval flows." | 细粒度：按类别（sandbox_approval_rules / skill_approval 等）逐项控制 |
| `never` | 须搭配 danger-full-access；否则回退到 read-only + approvals disabled | 从不请求审批 |

> 官方文档字符串列了 3 个值（untrusted / on-request / never），`granular` 在枚举中存在但可能较新。

### sandbox_mode（沙箱模式）

| 值 | 含义 |
|---|---|
| `read-only` | 不允许写操作 |
| `workspace-write` | 可写 cwd + writable_roots，越界需提权 |
| `danger-full-access` | 完全放开，无任何沙箱限制（须搭配 never） |

### 其他配置项

| 配置项 | 可选值 | 说明 |
|---|---|---|
| `collaboration_mode` | Default / Plan | Plan≈只可用 request_user_input，不执行修改 |
| `network_access` | true / false | 联网控制（[sandbox_workspace_write] 段） |
| `writable_roots` | 路径列表 | 额外可写路径 |
| `[features] codex_hooks` | true / false | 启用 .codex/hooks.json |
| `approvals_reviewer` | (auto) | "Approve for me"机制——自动批准提权请求。本质是"什么都不审批"=无脑放行 |

## 画像驱动机制（路线 A）

Codex 的画像来源是 `config.toml` 本身——`sandbox_mode` + `approval_policy` + `network_access` 三个值组合即唯一确定画像，无需额外的画像文件：

| sandbox_mode | approval_policy | network_access | → 画像 |
|---|---|---|---|
| read-only | never† | false | read |
| read-only | never† | true | search |
| workspace-write | on-request | true | work |
| workspace-write | on-request | true | auto |
| danger-full-access | never | true | bypass |

> † never + read-only 触发 Codex 回退到 "read-only + approvals disabled" = 真只读
> work 和 auto config 相同，区别在使用场景（有人/无人）

- **config.toml 即画像源文件**（等价于 Claude 侧读 payload 的 `permission_mode`）
- git pre-commit hook 可读 `config.toml` 拿 `sandbox_mode`/`approval_policy`，推导画像，再按画像调 L 级判定严度
- 用户切换画像 = 改 `config.toml` 对应三项（或用 `md/config.toml.模板` 中的预设注释段）

## 范围模型（in-scope / out-of-scope）

Codex 的"对话范围"= sandbox 允许写的路径：

| 概念 | Codex 实现 |
|---|---|
| 范围内（in-scope） | sandbox 允许写：`cwd` + `writable_roots` |
| 范围外（out-of-scope） | sandbox 拦截写：不在上述列表的路径 |

- **与 Claude 的 cwd 模型一致**：Claude 用 hook 读 payload 中的 `cwd` 判范围，Codex 用 sandbox 写路径判范围。两者范式一致（Codex 靠 sandbox、Claude 靠 hook+cwd）。
- bypass 画像下无"范围"概念——danger-full-access 不限制写路径。

## L0/L1/L2 × 画像 × 范围 三维写判定

### 范围内（sandbox 允许写，L 级判定由 git pre-commit + 安全审核.ps1 执行）

| 画像 | L0 | L1 | L2 |
|---|---|---|---|
| read | 硬 deny（写不了） | 硬 deny（写不了） | 硬 deny（写不了） |
| search | 硬 deny（写不了） | 硬 deny（写不了） | 硬 deny（写不了） |
| work | 双确认+审核 | 批准+审核 | 自由修改 |
| auto | 令牌+审核 | 自由修改 | 自由修改 |
| bypass | 自由修改 | 自由修改 | 自由修改 |

- read/search：sandbox=read-only → 所有写被 OS 拦截，approvals disabled → 连提权请求都不发 = 硬 deny
- work：工作区内 L2 自由、L1 需批准、L0 双确认
- auto：工作区内 L2/L1 自由、L0 需令牌（无人值守下替代交互双确认）
- bypass：全放开，但 S 级读禁地板仍在（check_s_level.ps1）

### 范围外（sandbox 拦截写，由 approval_policy 决定行为）

| 画像 | approval_policy | 范围外写行为 | 等价语义 |
|---|---|---|---|
| read | never | approvals disabled → 连提权请求都不发 | deny |
| search | never | 同上 | deny |
| work | on-request | 提权审批 → 用户批准 → 放行 → 再查 L 级 | ask |
| auto | on-request | 提权审批 → 无人应答 → 拦截 | deny* |
| bypass | never | 无范围概念（danger-full-access） | — |

> *auto 的 deny 不是 OS 硬拦，而是"请求了但无人批"的软拦。效果等同 deny。

- **Codex 天然支持"跨范围升 ask"**：`approval_policy = on-request` 时，范围外写触发提权审批（escalation），用户可批准/拒绝。Claude 用 hook 做 ask，Codex 用 sandbox 做 ask，行为等价。
- read/search 的"硬 deny"：`never` + `read-only` → approvals disabled → 连提权请求都不发 = 绝对 deny

### 与 Claude 3D 模型的对齐

| 维度 | Claude | Codex | 是否一致 |
|---|---|---|---|
| 范围来源 | hook 读 payload cwd | sandbox 写路径（cwd + writable_roots） | ✅ 一致 |
| 范围内 L 级×画像 | PreToolUse hook 判 | git pre-commit + 安全审核.ps1 判 | ✅ 行为一致 |
| 范围外行为 | hook 返 ask/deny | sandbox 提权(ask)/disabled(deny) | ✅ 等价 |
| L0 绝对保护 | 任何画像双确认 | 任何画像双确认+令牌 | ✅ 一致 |
| read 硬 deny | plan 模式（禁写+模型跳不出） | never+read-only（approvals disabled） | ✅ 对齐 |
| bypass 留 S 地板 | bypassPermissions + S hook | danger-full-access + check_s_level.ps1 | ✅ 对齐 |

## 令牌机制

auto 画像下（无人值守），L0 文件无法交互双确认。用令牌替代：

1. **预授权**：用户通过 `安全审核.ps1 -Approve -File <L0文件>` 预授权，生成一次性令牌写入审计日志
2. **检查**：git pre-commit 检查 L0 文件是否有有效令牌（未过期、未消费）
3. **消费**：commit 成功后令牌标记为已消费，不可重用
4. **令牌 = 安全审核.ps1 的 -Approve 记录**，无需额外令牌文件

- 令牌机制仅在 auto 画像下 L0 使用——替代交互双确认
- L1/L2 在 auto 画像下不需令牌（自由修改）
- work 画像下 L0 用交互双确认（人在场，直接对话确认）

## ask 语义

- `deny` → 任何情况硬拦、无窗（绝对）
- `ask` → 交互会话会弹窗等人批；只有真正无头（无人无窗定时任务）才穿透
- 人工批准层在交互下可用

**Codex 侧等价**：

| Claude 语义 | Codex 等价 | 机制 |
|---|---|---|
| ask | `approval_policy = on-request` 的提权审批 | Codex 呈现 justification → 用户批准/拒绝 |
| deny（硬拦） | `approval_policy = never` 的 approvals disabled | 连提权请求都不发 |
| deny（无人应答） | `on-request` 但无人批 | 请求发出但无人应答→软拦 |

## "Approve for me"问题（画像建立的根因）

Codex UI 的"Approve for me"（替我审批）通过 `approvals_reviewer` 实现自动批准——本质是**什么都不审批=无脑放行**。这不是"替我审批"，而是"放弃审批"。

正因如此，才需要建立画像系统来弥补：
- **auto 画像** ≠ "Approve for me"（不放弃审批，而是越界无人批=拦）
- auto = `on-request` + workspace-write → 范围内自由写、范围外需审批（无人→拦）
- 这才是"无人值守但仍有边界"的正确实现

## 文件层强制不对称（必须知晓）

- **Claude PreToolUse**：**编辑前**拦截（deny → 文件没被改）
- **Codex git pre-commit**：**提交前**拦截（文件已改但进不了 git 历史）
- 二者**不等价**。Codex 侧搭配 sandbox（防越界写）补足，但不能 100% 等价"编辑前拦截"。

---

## S/P 读判定（画像无关）

- **S 级**：拥有方 AI 可读写（任何画像下，含 bypass）；其他方 AI 不可读（任何画像下，含 bypass，`check_s_level.ps1` 硬拦 deny）
- **P 级**：所有 AI 可读（任何画像下）
- `check_s_level.ps1` **不读画像**——直接从 `S级清单.md` 读取对方 S 路径，命中即 deny
- **bypass 地板**：即使 bypass 画像全放开写权限，S 级读禁仍然生效（hook 由 `[features] codex_hooks` 控制，独立于 sandbox/approval 设置）
- bypass 模式下 hook 是否仍执行取决于 `[features] codex_hooks` 设置（部署后请自行确认所在环境实际行为）
- 详见 `md/hooks策略.md`

---

## S 级读取拦截

- Codex 侧：PreToolUse 匹配 `Bash` → `hooks/check_s_level.ps1` 拦截 S 级路径
- Codex sandbox 不支持 per-path denyRead
- bypass 画像下 S 读禁为地板级保护（部署后请自行确认 hook 在 bypass 模式下是否仍执行）
- 详见 `md/hooks策略.md`
